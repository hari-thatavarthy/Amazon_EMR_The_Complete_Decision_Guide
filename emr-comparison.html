Amazon EMR: Choosing Between EC2, Serverless, and EKS

A practical side-by-side comparison to help you pick the right deployment model for your workload.
Based on AWS documentation and Price List API data (us-east-1) · April 2026

You have a Spark job, a Hive query, or a Flink pipeline. You know you want to run it on Amazon EMR. The question that stops most teams is not whether to use EMR — it is which EMR.

AWS now offers three distinct ways to deploy EMR workloads, and they differ in almost every dimension that matters: what you manage, what you pay, how fast jobs start, which frameworks are available, and how far you can customize the environment. Picking the wrong one does not just cost more — it can lock your team into an operational model that fights your workload instead of serving it.

This guide puts all three options side by side across 40+ dimensions — architecture, frameworks, storage, scaling, pricing, security, operations, limitations, and real-world use cases — so you can make the decision once and make it well. Every data point is sourced from official AWS documentation and the AWS Price List API. No marketing fluff, no hand-waving, just the details you need.

1. Architecture and Compute Model

The most fundamental difference between the three EMR options is what you actually manage. EMR on EC2 gives you a full cluster of instances organized into node roles — you control instance types, storage, scaling policies, and the cluster lifecycle. EMR Serverless abstracts all of that away: you submit a job and AWS provisions, scales, and tears down compute behind the scenes. EMR on EKS sits in between — you manage an existing Kubernetes cluster, and EMR runs your analytics workloads as pods alongside your other containerized applications.

This architectural choice cascades into everything else: how fast jobs start, how large they can scale, how much operational overhead you carry, and what failure modes you need to plan for.

2. Framework and Runtime Support

This is often the first filter in the decision process. EMR on EC2 supports the broadest set of open-source frameworks — everything from Spark and Hive to HBase, Presto, Flink, Pig, and interactive tools like Zeppelin. EMR Serverless narrows the field to Spark and Hive. EMR on EKS supports Spark and Flink but not Hive or any of the legacy Hadoop ecosystem tools.

3. Storage

Storage architecture is one of the clearest differentiators. EMR on EC2 is the only option that gives you HDFS — a distributed file system running across the cluster's core nodes that provides high-throughput local I/O. This matters for workloads like HBase, heavy shuffle operations, or iterative ML algorithms that re-read data many times.

Both Serverless and EKS enforce compute-storage separation: all persistent data lives in Amazon S3, and compute resources scale independently. For the majority of modern Spark and Hive workloads, this is the recommended pattern regardless of deployment option. The question is whether your specific use case requires the local throughput that only HDFS provides.

Data persistence HDFS data lost on cluster termination; S3 data persists S3 only — always persists S3 only — always persists
Compute-storage separation Optional — you choose HDFS, S3, or both Enforced — S3 only Enforced — S3 only
4. Scaling and Elasticity

How well each option handles fluctuating demand has a direct impact on both performance and cost. EMR Serverless is the most elastic — workers spin up and down per-job, and the application can scale to zero when idle, meaning zero cost between jobs. EMR on EC2 offers managed scaling that evaluates cluster metrics every 5–10 seconds, but you always have at least one primary node running. EMR on EKS delegates scaling to Kubernetes tools like Karpenter, which can provision new nodes in seconds and drain them when empty.

The Spot Instance dimension is worth particular attention. Both EC2 and EKS support Spot pricing for up to 90% savings on compute, while Serverless does not offer Spot — though Graviton-based workers provide a roughly 20% discount as a partial offset.

5. Pricing

Each deployment option uses a fundamentally different pricing model, and the cheapest option depends entirely on your usage pattern. EMR on EC2 charges a per-instance-hour EMR surcharge on top of standard EC2 pricing — you can reduce the EC2 portion with Spot, Reserved Instances, or Savings Plans, but the surcharge is fixed. EMR Serverless rolls everything into per-second vCPU, memory, and storage rates — simple to predict, but no Spot discounts. EMR on EKS layers three costs: the EKS control plane, the underlying EC2 or Fargate compute, and an EMR surcharge per pod resource consumed.

The practical impact: for sustained 24/7 workloads, EC2 with Spot or Reserved Instances is cheapest. For intermittent workloads that run a few hours a day, Serverless wins because idle hours cost nothing. EKS is most cost-effective when the cluster is shared across analytics and non-analytics workloads.

6. Security and Access Control

All three options support the foundational security requirements: VPC isolation, encryption at rest and in transit, and IAM-based access control. The differences emerge in more advanced areas. EMR on EC2 offers the deepest security feature set — Kerberos authentication with Active Directory integration, Apache Ranger for policy-based authorization, and the most mature Lake Formation integration. EMR Serverless provides a simpler security model centered on runtime roles per job. EMR on EKS leverages Kubernetes-native constructs like namespaces, RBAC, and network policies for multi-tenant isolation.

Lake Formation integration is worth particular attention if you need fine-grained access control. Both EC2 and Serverless support the Fine-Grained Access Control mode (row, column, and cell-level filtering). Full Table Access mode — which is simpler and less restrictive — is available on EC2 and Serverless but not yet on EKS.

7. Operations and Customization

The operational burden varies dramatically. EMR Serverless is the lightest — you submit jobs and AWS handles provisioning, patching, and scaling. There is no infrastructure to maintain. EMR on EC2 gives you the most control but requires the most work: you manage cluster lifecycle, OS patching, YARN tuning, bootstrap actions, and scaling policies. EMR on EKS falls in the middle operationally — EMR manages the job pods, but you are responsible for the EKS cluster itself, including Kubernetes version upgrades, node group management, and monitoring.

Customization follows the same pattern. EC2 lets you SSH into nodes, run bootstrap scripts, use custom AMIs, and attach GPU instances. EKS allows custom container images and Kubernetes-native tooling. Serverless limits you to custom Docker images and Spark/Hive configuration overrides.

10. Decision Summary

If you take one thing away from this guide, it should be this: there is no universally correct answer. Each option exists because it serves a distinct operational model. The right choice depends on what frameworks you need, how your workloads behave over time, what infrastructure your team already manages, and how much operational responsibility you are willing to carry.

EMR on EC2

  • You need HDFS, HBase, Presto, or GPU instances
  • You run sustained 24/7 workloads and want RI or Spot pricing
  • You need the broadest framework support (12+)
  • You want SSH access and full OS-level customization
  • You require Kerberos or Apache Ranger
  • You need 3-primary-node high availability

EMR Serverless

  • You want zero infrastructure to manage
  • Workloads are intermittent, bursty, or event-driven
  • You only need Spark or Hive
  • Your team does not have infrastructure expertise
  • You want zero cost during idle periods
  • You need sub-second job startup via warm pools

EMR on EKS

  • You already run an EKS cluster for other workloads
  • You need Apache Flink for stream processing
  • You want Kubernetes-native tooling (ArgoCD, Karpenter, Helm)
  • You need multi-tenant isolation on shared infrastructure
  • You want to share compute between analytics and applications
  • You follow a GitOps workflow
